本仓库未定义专用错误类型体系或全局中间件，而是采用控制器内显式校验加路由层 try/catch 兜底的轻量模式。

1. 系统/框架
- 基于自定义 defineController 和 createRouter（lib/controller.ts）注册 REST 路由；路由处理器在匹配到路径后统一用 try/catch 包裹，捕获异常并返回 { success: false, error } 的 JSON 响应。
- 业务控制器通过 Response.json({ success, data|error }, status) 直接返回结构化响应体，由前端按字段区分成功失败。

2. 关键文件
- lib/controller.ts：路由注册、路径参数解析、全局错误兜底（500）、404 默认响应。
- controllers/*.controller.ts：各业务域控制器，负责入参校验与具体错误码（400/404/502）。
- lib/download-manager.ts：下载任务内部错误以状态机（pending/downloading/completed/failed/cancelled）加 progress.failed 计数表达，不抛错上抛。
- lib/epub-builder.ts：对非法输入使用 throw new Error(...)，由上层调用方处理。

3. 架构与约定
- 请求入口：Bun.serve -> createRouter() -> 匹配路由 -> try { handler(req) } catch -> 统一 500 JSON。
- 控制器职责：只做参数校验与业务调用；遇到可预期错误时返回带 success:false 的 Response，而非抛出异常。
- 后台异步任务（下载器）：将错误收敛为任务/章节级别的状态字段，并通过事件监听推送进度，避免跨进程边界抛错。
- 纯库函数（如 epub-builder）：对前置条件违反直接 throw Error，交由调用者决定如何包装为 HTTP 响应。

4. 开发者应遵循的规则
- 在控制器入口处校验必填参数，缺失时返回 Response.json({ success:false, error:"..." }, 400)。
- 不要从控制器中抛出未捕获异常；若必须抛错，确保外层 router 能将其转为 500 JSON。
- 对外部 I/O（网络、文件系统）使用 try/catch 包裹，并将错误信息放入 error 字段返回。
- 后台任务类（DownloadManager）使用状态机记录失败，不向 HTTP 层抛错。
- 共享库函数对非法入参使用 throw new Error(message)，保持调用链清晰。
